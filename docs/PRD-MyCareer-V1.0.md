## 1. 产品概述与愿景

### 1.1 产品定位

智能简历助手（MyCareer）是一款面向 0-3 年经验求职者的 AI 驱动职业发展平台，核心价值在于降低简历制作门槛、提升 JD 匹配效率、系统化准备面试。

### 1.2 用户核心痛点

| 痛点 | 描述 |
|------|------|
| 简历不会写 | 应届生缺乏经验，不知道如何将经历转化为 STAR 格式的简历语言 |
| JD 匹配迷茫 | 不确定自己的经历与目标岗位的匹配度，盲目投递效率低下 |
| 面试准备无章法 | 不知道面试官会问什么，缺乏答题框架和反馈 |
| 信息孤岛 | 简历、JD、面试、投递记录分散在多个工具中，无法关联管理 |

### 1.3 产品愿景

**让每位求职者都能高效、专业地呈现自己的能力，在竞争激烈的就业市场中脱颖而出。**

- 6 个月内成为目标用户群体（应届生/早期职场人）制作简历的首选工具
- 简历制作效率提升 70%（从平均 3 小时/份 → 45 分钟/份）
- 用户定制简历投递后的面试邀请率提升 25%

### 1.4 MVP 范围定义

本期 PRD 为 **V1.0 版本**，目标是修复已知缺陷、补齐核心链路的断点，并完善"投递记录"这一缺失的商业闭环功能。

## 4. 已完成功能基线

以下功能已通过测试验证，本 PRD 不对其核心逻辑做重大变更。

### 4.1 认证模块

- 邮箱 + 密码注册（bcrypt 哈希存储）
- 邮箱 + 密码登录（返回 JWT token，7 天有效期）
- Middleware 路由保护（公开路由：/login、/register；其余需认证）
- Cookie httpOnly + sameSite=lax

**成功指标**：注册转化率 ≥ 85%，登录成功率 ≥ 95%

### 4.2 简历管理

| 功能 | 描述 | 限制 |
|------|------|------|
| 上传解析 | 支持 PDF（≤10MB）和 Word（≤10MB） | 单文件，不支持批量 |
| 粘贴解析 | 用户粘贴原始文本，AI 解析为结构化数据 | 单次 ≤ 5000 字 |
| 标准简历生成 | 基于结构化数据，生成 STAR 格式中英双语简历 | 支持模板风格和长度选择 |
| 定制简历生成 | 基于 master resume + JD，生成针对性简历 | 需先有已解析的 JD |
| 来源追溯 | 每段经历标注来源原文和 AI 改写动作 | hallucinationRisk 等级 |
| 打招呼话术 | 基于定制简历生成 50-100 字打招呼短文 | 仅针对 tailored 类型简历 |
| AI 防护 | 确认项机制（inference/placeholder/quantification/keyword_align） | 待确认项必须处理后才能生成 |

**成功指标**：PDF 解析成功率 ≥ 90%，Word ≥ 85%，粘贴解析 ≥ 95%

### 4.3 JD 管理

| 功能 | 描述 |
|------|------|
| 粘贴 JD | 用户粘贴 JD 文本，AI 解析为结构化字段 |
| 结构化字段 | 技能权重（1-5）、硬技能/软技能/行业术语、职责、要求、加分项 |
| 匹配度分析 | matchScore（0-100）、逐项 skill 匹配状态、gap 分析 |
| 定制简历 | 基于 JD + master resume 生成定制简历 |
| 面试题生成 | 基于 JD 生成 8 道面试题 |

**成功指标**：JD 解析成功率 ≥ 92%，技能提取准确率 ≥ 75%

### 4.4 模拟面试

- 8 道固定题目结构：3 技术题 + 2 行为题 + 1 案例题 + 2 综合题
- AI 逐题评分（0-100）及分项反馈
- 整体反馈：知识点覆盖、回答思路、改进建议

**成功指标**：面试完成率 ≥ 60%，AI 评分满意度 ≥ 3.8/5

### 4.5 工作台

- 统计卡片：简历数量、JD 数量、面试练习次数、投递记录数
- 快捷入口：上传简历、粘贴 JD、开始面试

### 4.6 AI 防护体系

| 防护机制 | 实现方式 |
|---------|---------|
| 反幻觉 | hallucinationRisk 字段标注每条经历的 AI 推断风险等级 |
| 来源追溯 | provenance 数组记录原始文本、AI 改写动作、风险等级 |
| 确认项 | confirmableItems 数组，对不确定项必须用户确认 |

## 5. 问题与缺陷修复（本期 P0）

### 5.1 /resume/new 页面断链

**问题**：用户粘贴简历文本并完成 AI 解析后，系统跳转到 `/resume/new`，但该页面不存在，流程中断。

**修复方案**：
1. 新建 `/resume/new` 页面，承接 paste 解析结果
2. 展示 AI 解析的结构化简历数据，支持用户编辑确认
3. 提供生成选项（标准版/定制版简历），触发生成后跳转到简历详情页

**验收标准**：
- [ ] Paste 解析完成后能正常跳转到 /resume/new
- [ ] 页面展示解析结果，且用户可编辑
- [ ] 提交后生成简历并跳转到简历详情页

### 5.2 /jd/input 链接错误

**问题**：多处代码和导航中引用 `/jd/input` 路由，但实际页面路由为 `/jd/new`。

**修复方案**：全局搜索 `/jd/input`，统一替换为 `/jd/new`。

**验收标准**：
- [ ] 全局无 `/jd/input` 引用
- [ ] 所有 JD 相关入口链接指向 `/jd/new`

### 5.3 JD Status Schema 不一致

**问题**：TypeScript 类型定义与数据库实际可能值不一致。

```
TypeScript: "draft" | "parsed" | "tailoring" | "completed" | "archived"
Schema 可能：[draft, parsed, archived]  （缺少 tailoring / completed）
```

**修复方案**：
1. Audit 数据库现有文档，确认实际状态值
2. 迁移数据或统一类型定义
3. 添加数据库约束，防止非法状态写入

**验收标准**：
- [ ] TypeScript 类型与数据库实际状态值完全一致
- [ ] 新增状态变更必须经过类型校验

### 5.4 简历文件未上传至 CloudBase Storage

**问题**：用户上传的 PDF/Word 文件仅提取了文本内容，但原始文件未上传至 CloudBase Storage，`sourceFileId` 字段为空。

**影响**：用户无法下载原始文件，简历来源文件不可追溯，违反合规要求。

**修复方案**：
1. 集成 CloudBase Storage SDK
2. 简历上传时：先上传文件到 Storage，获取 fileId；再解析文本并存储 structured 数据
3. 更新 resume 对象的 `sourceFileId` 字段

**验收标准**：
- [ ] PDF 上传后 sourceFileId 字段有值
- [ ] Word 上传后 sourceFileId 字段有值
- [ ] 用户可下载原始上传文件

## 6. 新功能规划

### 6.1 投递记录模块（P1）

**问题背景**：用户生成定制简历和打招呼话术后，需要一个地方记录投递进度。当前面试"投递记录"仅为占位页"即将上线"，无任何 API 和 UI。

**功能范围**：

| 功能 | 描述 |
|------|------|
| 创建记录 | 选择关联的定制简历 + JD，填写公司名称、岗位、投递渠道、投递时间 |
| 列表页 | 按时间倒序展示所有投递记录，支持按状态筛选 |
| 状态更新 | 支持更新状态（pending → interview → offer/rejected） |
| 详情页 | 查看关联的简历、JD、面试次数、最终结果 |
| 删除/归档 | 软删除或归档不需要的记录 |

**数据结构**：

```typescript
interface Application {
  _id: string;
  userId: string;
  resumeId: string;         // 关联的 tailored resume
  jdId: string;             // 关联的 JD
  company: string;          // 公司名称
  position: string;         // 岗位名称（可从 JD 带入）
  channel: string;          // 投递渠道（BOSS直聘/官网/猎头等）
  appliedAt: Date;          // 投递时间
  status: "pending" | "interview" | "offer" | "rejected" | "archived";
  notes?: string;           // 备注（如面试反馈）
  interviewCount?: number;  // 面试次数
  result?: string;          // 最终结果
  createdAt: Date;
  updatedAt: Date;
}
```

**优先级理由**：投递记录是简历-JD-面试全链路的商业闭环节点。缺少此功能，用户在面试练习后无法关联真实的投递行为，数据断裂导致无法分析"简历版本-投递-面试"转化漏斗。

**成功指标**：
- 投递记录创建率：≥ 50%（完成定制简历的用户中，有投递记录的比例）
- 记录更新率：≥ 30%（创建记录后有状态更新的比例）

### 6.2 简历 PDF 导出（P1）

**问题背景**：当前用户可以生成标准版/定制版简历，但只能在线查看，无法导出为 PDF 文件。

**功能范围**：
- 一键导出当前简历为 PDF
- 支持中英文分页或合并
- 文件命名：`MyCareer_简历_{姓名}_{日期}.pdf`

**技术方案**：服务端 html2pdf（优先）或客户端 jsPDF（降级方案）

**验收标准**：
- [ ] 标准版简历可导出 PDF
- [ ] 定制版简历可导出 PDF
- [ ] PDF 内容与页面展示一致，中文显示正常

### 6.3 简历编辑页面（P1）

**问题背景**：简历 API 已支持 PATCH 更新，但无对应的编辑页面。用户解析简历后发现信息有误，只能重新上传或放弃。

**功能范围**：
- 编辑基本信息（联系人）
- 编辑工作经历（增删改 STAR 描述）
- 编辑项目经历（增删改描述）
- 编辑教育背景
- 编辑技能标签
- 保存后重新生成简历内容

**交互设计要点**：
- 在线编辑 + 实时预览（类似 Notion 体验）
- AI 辅助优化：用户写一段描述，AI 提供 STAR 格式优化建议
- 变更 diff：编辑前后对比

**验收标准**：
- [ ] 五大模块（contact/experiences/projects/education/skills）均可编辑
- [ ] 保存后 structured 数据正确更新
- [ ] 编辑后的简历可重新生成标准版/定制版

### 6.4 简历归档功能（P2）

**问题背景**：简历状态机已定义 `archived` 状态，但无 UI 操作入口。用户积累多份简历后，无法有效管理。

**功能范围**：
- 列表页增加"归档"操作
- 归档后的简历移至"归档"Tab，不影响主列表
- 支持从归档恢复

**优先级理由**：归档是低频需求，不影响核心链路，延后至 V1.1。

### 6.5 英文 i18n 补全（P2）

**问题背景**：英文翻译文件（messages/en.json）基本为空，切换语言后页面显示不完整。

**功能范围**：
- 补全所有中文 key 的英文翻译
- 术语一致性（简历领域专业词汇：STAR/achievement/qualification 等）
- 验证所有页面无硬编码中文

**优先级理由**：英文支持是加分项而非本期 MVP 核心，当前目标用户为中文母语者，V1.0 可延后。

### 6.6 优势识别（P1）

**问题背景**：用户完成简历解析后，往往对自己的职业方向仍然迷茫——他们有技能，但不知道这些技能可以迁移到哪里。当前功能（简历生成/JD匹配）都是"定向"工具，用户已有明确目标岗位。优势识别是"发散"工具，帮助用户发现未曾考虑的可能性，突破职业思维局限。

**功能定位**：
- 独立入口，不依赖简历或 JD，可作为用户首次使用时的"自我探索"环节
- 与现有功能形成互补：先发现方向 → 再生成简历 → 再匹配 JD → 再练习面试 → 最后记录投递

#### 用户故事

**Story 1**：作为求职者，我有时不确定自己还能做什么，希望系统能基于我的技能和兴趣，推荐一些我从没想过的职业方向，帮助我打破思维局限。

**Story 2**：作为求职者，我想要通过回答几个简单的问题，获得对自己核心优势的清晰认知，而不是看一堆模糊的"性格测试"结果。

**Story 3**：作为求职者，我想要具体的、可执行的职业探索路径，不是泛泛的"你要多尝试"，而是有明确的第一步。

---

#### 问卷设计

**核心原则**（5条）：
1. **最小认知负荷**：控制在 5-7 个问题，每个问题不超过 3 个选项（文本题除外），使用渐进式披露
2. **成就行销**：每个问题让用户感觉"我有东西可以说"，通过"心流体验"和"成就事件"间接定位内在动机，而非直接问"你会什么"
3. **零评判安全感**：选项设计去职业歧视化，不设"正确"选项，避免触发防御性反应
4. **身份认同锚定**：问题设计持续提供"你是一个做XX事的人"的身份暗示，帮助用户在回答过程中逐步建立自我叙事
5. **认知留白**：某些问题设计为"选填"，给用户控制感，降低"必须准备好所有答案"的压力

**问题流程**：

| # | 问题 | 形式 | 目的 |
|---|------|------|------|
| 1 | 你现在处于哪个阶段？ | 单选 | 职业状态基线，建立个性化起点 |
| 2 | 最近一次觉得"时间过得很快、做得很投入"是什么时候？ | 卡片多选（最多3个） | 通过心流体验间接定位内在驱动力 |
| 3 | 以下哪件事最让你有成就感？ | 单选 + 可选文本 | 探测成就动机类型（搭建造物/说服他人/解决问题/培养他人/竞争获胜/创意认可） |
| 4 | 你理想中的工作环境是什么样的？ | 5维度量表（1-5分） | 获取工作环境偏好，过滤推荐结果 |
| 5 | 以下三件事，如果只能选一个长期拥有，你选哪个？ | 强制排序（6项） | 探测核心工作价值观（收入/助人/成长/影响力/平衡/挑战） |
| 6 | 想象你需要从零开始学习一个新领域，可能半年内收入降低，你会怎么做？ | 情境题（单选） | 探测职业转型风险承受力 |
| 7（选填） | 你通常是怎么学会一件新事物的？ | 多选（最多2个） | 探测学习风格，为后续资源推荐提供依据 |

**心理学改进**（基于心理学家审查）：
- Q1 拆分为两个独立维度：就业状态 + 职业清晰度，避免构念混淆
- Q3 避免社会赞许性偏差："解决了别人解决不了的问题"改为间接行为探测，减少自我提升偏差
- Q5 强制排序 6 项认知负荷较高，在选项前增加引导语"没有标准答案，根据直觉排序"
- Q6 增加情境约束："假设你有一份稳定工作，但有一个你感兴趣的新领域机会"，避免不同用户对"风险"理解不一致
- 增加全局一致性检验：在 Q3 与 Q6 之间插入反向表述的同一维度题，防止用户"乱答"

**交互设计要点**：
- **单页单题卡片式**：每次只展示一个问题，减少认知分散
- **进度条赋义**：不用"3/7"，而是"你已经发现了你的动力来源"
- **Q3→Q4 缓冲插入**：连续两个高认知负荷问题之间插入喘息文案："休息一下——接下来的问题关于你理想的工作方式，不是关于你的过去"
- **退出挽留**：引用用户已回答内容制造损失感，而非"确定要退出吗"这种废话
- **结果预览钩子**：Q4 完成后展示"软性线索"（如"你的核心优势可能与分析解决问题相关"），增加完成动力
- **智能预填充**：若用户已有简历，从简历提取技能关键词预填充，用户只需确认或补充

#### AI 输出要求

基于用户回答，输出以下内容：

| 输出模块 | 描述 |
|---------|------|
| 可迁移技能总结 | 识别 3-5 个核心可迁移技能，说明从当前背景迁移到目标方向的依据 |
| 意想不到的职业道路 | 5-7 个出人意料但现实可行的职业方向，含行业、入门途径、薪资参考（注明"仅供参考"）、搜索策略 |
| 快速取胜的机会 | 探索这些路径的 3 个直接步骤 + 资源推荐 |
| 现实检验 | 最匹配的路径 + 各阶段过渡的预期时间表 |

**约束**：
- 薪资范围必须注明"仅供参考，来源于公开数据，不构成就业承诺"
- 投递方式提供搜索策略，不提供具体链接（避免过时）
- 不知道就说"暂无公开数据"，不编造

## 7. 技术约束与依赖

### 7.1 环境限制

| 约束项 | 说明 |
|--------|------|
| 运行时 | CloudBase 云函数（Node.js），非 Edge Runtime |
| 图片优化 | 无 Sharp，无法服务端图片压缩/裁剪 |
| 无头浏览器 | Standalone 模式，原生支持 headless Chrome（如需 PDF 生成需额外调研） |
| 文件存储 | CloudBase Storage，API 方式上传 |

### 7.2 外部依赖

| 依赖 | 用途 | 稳定性 |
|------|------|--------|
| DeepSeek API | 所有 AI 能力（简历解析、JD 解析、面试题生成、评分） | 依赖方，无备选 |
| CloudBase DB | 数据库（MongoDB 兼容） | 稳定 |
| CloudBase Storage | 文件存储（简历 PDF/Word） | 稳定 |
| CloudBase Auth | 用户认证 | 稳定 |

### 7.3 安全约束

- JWT 7 天过期，需要 refresh token 机制或引导重新登录
- 文件上传 ≤ 10MB，前端校验
- 所有 API 必须校验 userId，防止越权访问
- 敏感操作（删除简历、删除 JD）需二次确认

### 7.4 性能目标

| 指标 | 目标值 |
|------|--------|
| 页面首次加载（LCP） | ≤ 3 秒 |
| API 响应时间（P95） | ≤ 2 秒 |
| 简历解析耗时 | ≤ 10 秒 |
| JD 解析耗时 | ≤ 5 秒 |
| AI 面试评分 | ≤ 5 秒/题 |

## 8. 上线计划与里程碑

### 8.1 V1.0 上线计划（目标：2026-Q3）

| 阶段 | 内容 | 目标日期 | 里程碑 |
|------|------|---------|--------|
| **Phase 0：优势识别** | 问卷前端 + AI 集成 + 测试 | Week 1-2 | 用户获取钩子，导流至简历生成 |
| **Phase 1：缺陷修复** | 修复 /resume/new 断链、/jd/input 链接错误、JD status 不一致、文件上传 | Week 2-3 | 核心链路贯通 |
| **Phase 2：投递记录 MVP** | API CRUD + 列表页 + 状态更新 | Week 4-5 | 商业闭环补全 |
| **Phase 3：PDF 导出 + 编辑页面** | PDF 导出 + 简历编辑 UI | Week 6-7 | 功能完整度提升 |
| **Phase 4：灰度上线** | 10% → 50% → 100% 用户 | Week 8 | GA |

> 注：Phase 0 优先上线，因优势识别是独立功能，且可作为新用户首次访问的引导入口，早于简历生成功能出现，有效提升用户激活率。

### 8.2 灰度策略

- **第一周**：10% 用户，观察核心指标（解析成功率、转化率）
- **第二周**：50% 用户，扩大样本，监控性能指标
- **第三周**：100% 用户，全面上线

### 8.3 回滚条件

| 指标 | 阈值 | 动作 |
|------|------|------|
| 简历解析成功率 | < 80% | 回滚本期代码 |
| API 错误率 | > 5% | 回滚本期代码 |
| P95 响应时间 | > 5 秒 | 排查优化或回滚 |
| CloudBase 账单异常 | 超出预算 20% | 紧急扩容 + 账单告警 |
